# FAQ

## Общие

### Работает ли с MiniShop2?

Нет. Только **MiniShop3** и MODX 3.

### Нужен ли VueTools?

Да, для SPA менеджера. На витрине VueTools не требуется.

### Есть ли своё оформление заказа?

Нет. Покупатель оформляет заказ как обычный товар MS3.

## Установка и SPA

### SPA не открывается

1. Установлен **[VueTools](/components/vuetools/)** 1.1.2-pl+.
2. Файлы `assets/components/premiumaccess/js/mgr/vue-dist/premiumaccess-admin.min.{js,css}` на месте.
3. Консоль браузера в менеджере без ошибок Import Map.
4. Кэш MODX очищен.

### Нет пункта меню PremiumAccess

Переустановите пакет PremiumAccess или проверьте пункты меню в resolver. У пользователя менеджера должно быть право **`view`**.

## Paywall и контент

### Контент не закрывается

1. **Правило** включено для страницы или группы.
2. Плагин **`premiumaccess_content_protection`** включён.
3. Сниппеты **некэшируемые** (`[[!...]]`).
4. **`premiumaccess.enabled`** = Да.
5. Кэш MODX очищен.

### Premium block не скрывается

1. UUID в маркере совпадает с block в **Premium-блоки**.
2. Правило с типом `premium_block` на этот UUID.
3. Маркер в content: `[[pa-block:uuid]]` без опечаток.

### Paywall «залипает» под другим пользователем

Кэш HTML (например ImageOptimizer). Очистите кэш. Шаблоны `premiumaccess_*` выставляют `imageoptimizer.skip_html_cache`; авторизованным пользователям кэш не отдаётся.

## miniShop3 и доступ

### Доступ не выдаётся после оплаты

1. **`premiumaccess.enabled`** = Да.
2. **`premiumaccess.paid_order_statuses`** совпадает со статусом «Оплачен» заказа.
3. В тарифе PremiumAccess указан **тот же ID товара**, что в позиции заказа.
4. Плагин **`premiumaccess_order_access`** включён на **`msOnChangeOrderStatus`**.
5. Заказ привязан к пользователю web-контекста (`user_id` / `createdby`).

### Отмена заказа не отзывает доступ

Проверьте список **отменённых статусов** в настройках (`premiumaccess.revoked_order_statuses`). Отзывается только доступ, выданный **по этому заказу**. Ручная выдача и промокод не отзываются.

### Выдача после оплаты есть, контент закрыт

В **Доступы** запись есть, но страница закрыта:

1. На **эту** страницу повешено правило с **тем же тарифом**, что выдан пользователю (тариф в правиле и в выдаче должны совпадать).
2. Доступ **активен** — не истёк и не отозван.
3. Очистите кэш MODX (и кэш отказов PremiumAccess, если меняли правила недавно).
4. Проверьте под тем же пользователем через `web/access/check`.

### Статус «Новый» заказа выдал доступ

ID статуса «Новый» попал в `paid_order_statuses`. Оставьте только статус «Оплачен» (`ms3_status_paid`, обычно `3`).

## Проверка доступа и правила

### В ответе API: доступ запрещён, причина «правило не найдено»

На страницу или файл **не повешено правило**. Привяжите тариф в **Ресурсы** или создайте правило в **Правила**.

### Несколько тарифов на одну страницу

Можно создать несколько правил с разными тарифами. Достаточно **одного** активного доступа на любой из них.

### Бессрочный vs период

Задаётся в тарифе (**Продукты** → тип срока). Дату окончания можно изменить в **Доступы** → **Продление** или редактирование дат.

## Защищённые файлы

### Скачивание файла: ошибка 403

1. Правило типа «файл» + **активный доступ** к тарифу из правила.
2. Файл существует в **`premiumaccess.protected_path`**.
3. Token не истёк (`download_token_ttl`).
4. CSRF: `modAuth` в форме `paFileDownload`.

### Файл доступен по прямому URL

Перенесите хранилище вне document root. Закройте каталог на web server.

## Промокоды

### Промокод не активируется

1. Пользователь **вошёл** на сайт (не гость).
2. Код **активен**, срок не истёк, лимит активаций не исчерпан.
3. Этот пользователь **ещё не активировал** этот код раньше.
4. После серии неудачных попыток подождите минуту.

## Уведомления

### Email не уходит

1. **`premiumaccess.notifications_enabled`** = Да.
2. SMTP MODX настроен.
3. Плагин **`premiumaccess_notifications`** включён.
4. Chunks `paEmailGrant*` установлены.

### Cron истечения не пишет в журнал

1. **`premiumaccess.expire_cron_enabled`** = Да.
2. Есть выдачи доступа с прошедшей датой окончания (`expires_at`) и без даты отзыва (`revoked_at`).
3. Запустите CLI: `php core/components/premiumaccess/bin/expire-accesses.php`.
4. Проверьте stdout: `expired=N, auto_renewal_reminders=N`.
5. Если скрипт уже запущен, повторный запуск ничего не сделает (код **0**) — подождите и попробуйте снова.

### Webhook не приходит

1. `premiumaccess.notify_webhook_url` заполнен.
2. `notifications_enabled` = Да.
3. URL принимает POST с JSON и отвечает успехом (HTTP 2xx).
4. Ошибки в MODX error log `[PremiumAccess]`.

## Кастомизация

### Свой плагин на выдачу

Подпишитесь на **`OnPaAccessGrant`**. Не выдавайте доступ из своего плагина на смену статуса заказа — компонент при повторной оплате не создаёт второй доступ и событие не дублирует.

См. [События MODX](development/events).

### Свой интерфейс на витрине

Для AJAX без серверных чанков используйте connector: `web/access/check`, `web/cabinet/list`, `web/access/renew-link`. Справочник actions: [Connector API](development/api). Для PHP в своих сниппетах — [События MODX](development/events).
